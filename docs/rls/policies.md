# RLS ポリシー

## この資料の役割

`supabase/migrations/0001_initial.sql` が機械可読な正本であり、この資料は **人間向け補足** です。レビュー時には、SQL とこの説明を両方見ます。

## 基本方針

- 認証済みユーザーだけがアプリデータへアクセスできる
- `profiles` は全認証済みユーザーが参照可能
- `tasks` は認証済みユーザーが参照可能、作成者または admin が更新可能
- `comments` は認証済みユーザーが閲覧・投稿可能
- `labels` は認証済みユーザーが参照可能
- `task_labels` は `tasks` と同じ方針に従う

## 教材として重要な観点

### UI ガードと RLS は役割が違う

- UI ガード: 画面を見せないための親切な導線
- RLS: そもそもデータベース操作を通さないための安全装置

### なぜ RLS を docs に残すか

SQL だけでは意図が伝わりにくく、レビューで「なぜその条件なのか」が抜けがちです。授業では、以下を説明できる状態を目指します。

- どのロールが何をできるか
- その条件を DB で守る理由
- アプリ側の分岐だけに頼ると何が危険か
